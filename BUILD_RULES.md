# Aristolegion Build Rules

Version: 1.0

Status: LOCKED

---

## Purpose

This document defines the permanent engineering rules for the Aristolegion website.

All future development must comply with these rules.

---

# Philosophy

Aristolegion is NOT a startup.

It is NOT a SaaS.

It is NOT a portfolio.

It is a modern intellectual institution.

Every design decision should reinforce:

- Authority
- Elegance
- Timelessness
- Intellectual depth
- Premium craftsmanship

---

# Design References

Primary inspirations

- Stripe Press
- Farnam Street
- Harvard Business Review
- McKinsey Insights
- Apple
- Linear
- Notion

Avoid startup aesthetics.

Avoid flashy gradients.

Avoid excessive animations.

Avoid bright colors.

---

# Visual Identity

Primary Colors

- Navy
- Gold
- Ivory

Typography

- Display serif for headings
- Modern sans for body

Spacing

Generous whitespace.

Content must breathe.

Never overcrowd sections.

---

# Development Principles

Always build production-ready code.

Never create placeholder implementations.

Never duplicate components.

Prefer reusable components.

Prefer composition over repetition.

Every component must be responsive.

Accessibility is mandatory.

SEO is mandatory.

Performance is mandatory.

---

# Code Standards

Use

- Next.js App Router
- TypeScript
- TailwindCSS

Avoid inline styles.

Avoid hardcoded colors.

Avoid unnecessary libraries.

Keep components small.

Keep content separated from UI.

---

# UX Principles

Every page should feel like reading a premium publication.

Scrolling should feel calm.

Animations should be subtle.

Hover effects should communicate quality.

Nothing should feel rushed.

---

# Mobile

Desktop-first.

Then optimize for:

- Tablet
- Mobile

Every page must remain beautiful on all screen sizes.

---

# Images

Use Next/Image.

Optimize all assets.

Lazy load wherever appropriate.

Maintain consistent aspect ratios.

---

# Copywriting

Writing should sound:

- Intelligent
- Calm
- Authoritative
- Timeless

Never use marketing hype.

Never use clickbait.

Never use startup jargon.

---

# Git Workflow

Every completed milestone must be:

Commit

Push

Verified

before beginning the next milestone.

---

# PRD Compliance

The PRD is the source of truth.

If implementation conflicts with the PRD,

the PRD always wins.

---

# Completion Rule

No feature is considered complete until:

- responsive
- accessible
- production ready
- polished
- reviewed
